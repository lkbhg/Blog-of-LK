## 💡 背景与问题描述

在使用 Nekoray 或以 sing-box 1.9.7 为核心的代理工具时，你可能遇到过一个非常令人崩溃的问题：**在打开 TUN 模式的情况下，只要稍微更改与 TUN 相关的设置（如 DNS、Route 等），就会导致 TUN 模式永久性失效。** 此时，日志里会疯狂报错 `dns exchange failed`。

奇怪的是，如果直接删除现有的 Nekoray 文件夹并重新解压一份，TUN 就能奇迹般地恢复正常。

此外，在开启 `TUN-IPv6` 后，还会出现一种“半残”状态：

* **部分网站无法访问**（例如 ChatGPT）。
* **部分网站正常访问**（例如 Google 搜索）。
* 日志显示核心错误：`dial tcp6: network unreachable`。

经过深入排查，我将问题的根因锁定在**路由表回环错误**、**DNS 域名解析死循环**以及**代理服务器对 IPv6 的支持不完善**上。

## 📊 可能的两种错误

| 错误表现 | 核心报错日志 | 触发场景 | 根本原因 |
| --- | --- | --- | --- |
| **错误 A：TUN 彻底失效** | `dns exchange failed` | 更改 DNS/Route 配置后发生，只能重装或删文件夹恢复。 | **路由表污染无法回退** + DNS 死循环。 |
| **错误 B：部分网站无法访问** | `dial tcp6: network unreachable` | 开启 TUN-IPv6 后，ChatGPT 等网站挂掉，但 Google 正常。 | **节点不支持 IPv6**，但 TUN 强行接管了 IPv6 流量。 |

## 🔍 一、 现象分析与 TUN 工作原理

要彻底解决问题，首先需要了解 TUN 模式在 Windows 下的流量接管流程：

> **系统流量 → Wintun 虚拟网卡 → Mihomo/Clash/Singbox 内核 → DNS 解析 → 代理出口**

在这个链条中，只要出现 **DNS 服务器不可达**、**路由规则写错**或**默认路由被覆盖**，就会直接触发 `dns exchange failed`。

---

## ⚠️ 二、 导致 TUN 彻底卡死的“三大致命点”

为什么随便改个设置，TUN 就会永久失效？核心原因在于**DNS 死循环**与**路由表污染**。

### 🔴 致命点 1：DNS 解析死循环
在 Windows 系统中，TUN 模式的工作原理是接管系统的“默认网关”（如 `0.0.0.0/0` 和 `::/0`），将所有流量强制引流到 Wintun 虚拟网卡中。[根本原因：路由表污染与回退失败](#-根本原因路由表污染与回退失败)

如果配置了 `dns_final_out = proxy`，意味着 DNS 请求本身也要走代理。这就形成了一个死结：

1. 请求解析 `dns.google` ➡️ 需要 DNS。
2. DNS 请求 ➡️ 走 Proxy。
3. Proxy 建立连接 ➡️ 依然需要 DNS。
**结果：无限循环 ➡️ exchange failed。**

### 🔴 致命点 2：错误使用 DoH (DNS over HTTPS)

如果你使用的是 DoH URL（如 `https://dns.google/dns-query`）而非单纯的 IP 地址，DoH 在工作前首先需要解析出 `dns.google` 的 IP。如果此时全局流量被代理，且没有正确配置 bypass（直连）规则，就会直接卡死。

### 🔴 致命点 3：sing-box 1.9.7 的路由残留

在 Windows 下开启 TUN 的 IPv6 时，分流极不稳定（特别是在有 ZeroTier、多网卡、Wi-Fi Direct 的复杂网络环境中）。
如果关闭了 `strict-route`，sing-box 在退出或重启时，**无法自动找回旧路由**，也不会重建原有的 IPv6/IPv4 网关。
这就是为什么“重新解压软件”能解决问题：因为删除程序并重启，强制系统刷新了网卡状态，清空了被污染的路由表。

## 🛠️ 三、 终极解决方案

## 💥 错误 A：更改配置导致 TUN 永久失效 (DNS/路由死锁)

**症状：**
只要在开启 TUN 模式的情况下，修改了 DNS、路由规则等设置，TUN 就会“永久性卡死”。所有网页无法打开，日志狂刷 `dns exchange failed`。删除现有的 Nekoray 并重新解压一份，又奇迹般地恢复了。

### 🔍 根本原因：路由表污染与回退失败

在 Windows 系统中，TUN 模式的工作原理是接管系统的“默认网关”（如 `0.0.0.0/0` 和 `::/0`），将所有流量强制引流到 Wintun 虚拟网卡中。

**为什么无法回退？**

1. **重载机制缺陷：** 当你在 sing-box 运行期间更改配置，内核会尝试重新初始化 TUN 设备或重载网络栈。但在某些情况下（尤其是关闭了 `strict-route` 时），sing-box 在销毁旧的 TUN 会话时，**未能向 Windows 操作系统发送正确的撤销路由指令**。
2. **残留的“幽灵路由”：** 旧的默认路由依然留在 Windows 的路由表中，指向一个已经失效或被重新分配的 Wintun 虚拟网卡接口。
3. **彻底死锁：** 此时你产生的新流量，全部被系统丢弃到了那个失效的“幽灵路由”里，根本进不了新的代理内核，从而导致 TUN 彻底失效。
4. **为何重装能好？** 删除整个文件夹并重新运行，实际上是迫使软件重新向系统申请并写入全新的网卡状态，覆盖了之前的错误配置。

### 🔍 附带死因：DNS 解析死循环

如果你的配置中写了 `dns_final_out = proxy`，并且使用了 DoH（如 `https://dns.google/dns-query`）：

* 系统需要解析 `dns.google` ➡️ DNS 请求被路由表丢进代理 ➡️ 代理需要连接服务器，又需要 DNS ➡️ 触发无限死循环 ➡️ `dns exchange failed`。

### 🛠️ 解决方案（针对错误 A）

**1. 紧急清空路由表（无需重装软件）：**
当 TUN 崩溃时，以管理员身份运行 CMD，强制清理被污染的路由表：

```cmd
route -f
ipconfig /flushdns

```
*⚠️ 注意：此命令会清空所有路由，执行后**务必重启电脑以重建正常的本地路由表（或重启网卡，但通常还是需要重启电脑）**。*

**2. 优化 sing-box 配置：**

* **推荐开启**：`auto_route: true` 和 `strict_route: true`（让内核严格控制和恢复路由）。
* **使用 Fake-IP**：避免复杂的 DoH 导致的死循环，直接使用基础 IP 作为 DNS 服务器（如 `223.5.5.5`, `8.8.8.8`），并将模式设置为 `fake-ip`。

## 🕳️ 错误 B：开启 TUN-IPv6 后的“网络半残”现象

**症状：**
开启 TUN 模式的 IPv6 接管后，Google 搜索可以正常使用，但访问 ChatGPT 或部分特定网站时疯狂转圈，日志报错 `dial tcp6: network unreachable`。

### 🔍 根本原因：IPv6 流量“有去无回”

这个问题与路由表卡死无关，纯粹是**链路支持不完整**导致的：

1. **TUN 强行接管：** 你在配置中开启了 `"ipv6": true`，TUN 网卡生成了 IPv6 的默认路由 `::/0`，系统将所有的 IPv6 流量都交给了代理内核。
2. **节点不支持：** 但现实是，你的代理服务器（通过节点域名订阅）根本没有配置 IPv6 地址（没有 AAAA 记录）。
3. **触发断网：** 当你访问优先使用 IPv6 的网站（如 ChatGPT）时，流量进入了 TUN，但代理节点无法用 IPv6 把请求发出去，直接抛出 `network unreachable`。
4. **为何 Google 能上？** 因为 Google 的服务端和某些浏览器机制会自动快速 Fallback（回退）到 IPv4，掩盖了这个错误。

### 🛠️ 解决方案（针对错误 B）

测试你的节点是否支持 IPv6（CMD 输入：`nslookup -type=AAAA 你的订阅域名`）。如果没有返回 IPv6 地址，请从以下方案二选一：

**方案 1：直接关闭 TUN IPv6 接管（最稳妥）**
既然代理不支持，就不要让 TUN 瞎凑热闹。在配置中明确关闭它，让系统原生网卡处理本地 IPv6 流量。

```json
"ipv6": false

```

**方案 2：强制全局域名解析优先 IPv4**
如果你必须开启 IPv6，请在 sing-box 的 DNS 策略中强制走 IPv4，避免因为请求 IPv6 资源而撞墙：

```json
"domain_strategy": "prefer_ipv4"

```
### 🛡️ 调整 sing-box / Nekoray 核心配置（长效解决）

根据节点情况，选择以下三种配置策略之一：

#### 策略 A：关闭 TUN IPv6 接管（最稳妥，强烈推荐）

如果你的代理节点本身不支持 IPv6，强行接管只会徒增烦恼。让系统默认处理 IPv6，代理处理 IPv4 即可。

```json
"ipv6": false
```
#### 策略 B：强制所有流量优先走 IPv4

在 sing-box 的路由配置中添加以下设置，强制域名解析优先使用 IPv4，避免因为请求 AAAA 记录导致超时：

```json
"domain_strategy": "prefer_ipv4"
```
#### 策略 C：实现 IPv6 全接管（仅限节点支持 IPv6）

如果你确定节点支持 IPv6，可以开启全接管。
**测试节点是否支持 IPv6：**

* `nslookup -type=AAAA 你的订阅域名` （查看是否有记录）
* `curl -6 https://你的订阅域名` （测试是否连通）

如果支持，请务必开启 `auto_route` 并严格规范配置：

```json
{
  "type": "tun",
  "stack": "system",
  "auto_route": true,
  "strict_route": false,
  "ipv6": true,
  "inet4_address": "172.19.0.1/30",
  "inet6_address": "fdfe:dcba:9876::1/64"
}
```
## 📌 四、 防坑指南与建议

1. **DNS 推荐设置**：在 TUN 模式下，强烈建议使用 Fake-IP 模式。DNS 服务器填基础 IP（如 `223.5.5.5` 或 `8.8.8.8`）。**不要乱填 DoH URL**，除非你非常清楚如何配置 fallback 和直连路由。**当然，直接把DNS路由关掉也行**。
2. **避免代理冲突**：不要混用“系统代理（System Proxy）”和“TUN 模式”，这两者同时开启极易引发流量回环。
3. **清理幽灵网卡**：如果经常报错，打开 `控制面板 → 网络适配器`，或者在设备管理器中勾选“显示隐藏设备”，删除多余残留的 `Wintun` 虚拟网卡。

> **总结**：在 Windows 复杂的网络环境下，TUN 的 IPv6 默认路由极其脆弱。**“如果没有明确的 IPv6 代理需求，关闭 TUN 模式下的 IPv6 接管是保持身心健康的不二法则。”**
